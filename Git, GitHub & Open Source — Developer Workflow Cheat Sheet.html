Git Basics

Your Daily Git Workflow

Write Code→ git status→ git add→ Staging Area→ git commit→ Local Repository→ git push→ GitHub

1 · Configure Git

Tell Git who you are — done once per machine.

bash
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

2 · Create or Clone a Repo

Start fresh, or copy an existing project locally.

bash
git init                      # start a new repo
# or
git clone <repository-url>    # copy an existing one

3 · Daily Workflow

The loop you'll repeat dozens of times a day.

bash
git status
git diff
git add .
git commit -m "Add user authentication"
git push -u origin main        # -u only needed the first time

4 · Branch Workflow

Isolate new work from main so it stays shippable.

bash
git switch -c feature/user-auth
# older equivalent:
git checkout -b feature/user-auth

From Feature Branch to Pull Request

main→ feature branch→ write code→ test→ commit→ push→ Pull Request→ Code Review→ Merge

Common Hiccups

Remote rejected

bash
git pull --rebase origin main
git push origin main

Unstage a file

bash
git restore --staged <file>

Discard changes

bash
git restore <file>

Merge conflict

bash
# fix the conflict manually
git add .
git commit -m "Resolve merge conflict"
git push origin main
⚠️ Be careful with git reset --hard — it permanently discards uncommitted work.
Staying In Sync

Remotes, Fetch & Merge

git pull is really just git fetch + git merge in one step. Here's what's happening underneath, plus a few remote commands worth knowing.

See Your Remotes

List them, then refresh tracking info for all of them at once.

bash
git remote -v
git fetch --all

Fetch + Merge = Pull

Two steps you can also run separately, when you want to inspect before merging.

bash
git fetch upstream
git merge upstream/main
# same result as:
git pull upstream main

Push to a Named Branch

Handy when your local branch name doesn't match the remote.

bash
git push origin HEAD:fix/typo
Open Source

Your First Open Source Contribution

Find Project→ Read README→ Check CONTRIBUTING.md→ Find Issue→ Fork→ Clone→ Create Branch→ Code + Test→ Push→ Pull Request→ Review→ Merge
💡 Before opening an issue: search existing issues first so you're not filing a duplicate. When you do file one, include a clear summary, exact steps to reproduce, and a screenshot if it's visual — maintainers can act on it much faster.
bash
git clone <your-fork-url>

git remote add upstream <original-repository-url>
git fetch upstream

git switch main
git pull upstream main

git switch -c fix/documentation

# make changes

git add .
git commit -m "Improve documentation"

git push origin fix/documentation

Two Remotes, One Repo

originYour GitHub fork
upstreamThe original project

Keeping upstream synced means your fork — and your pull requests — never drift far from the real project.
